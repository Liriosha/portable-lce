6// 4J added - Storage for data (ie the extra per tile storage). Data is normally
7// stored as 4-bits per tile, in a DataLayer class of 16384 bytes ( 128 x 16 x
8// 16 x 0.5 ) This class provides more economical storage for such data by
9// taking into consideration that it is quite common for large parts of the data
10// to be very compressible (ie zero). We are aiming here to balance performance
11// (this data is accessed very frequently) against size.
12
13// Details of storage method:
14
15// 1. Data is split into horizontal planes, of which there are 128, and each
16// taking up 128 bytes (16 x 16 x 0.5)
17// 2. Each of these layers has a permanently allocated index in this class
18// (planeIndices).
19// 3. Data for allocatedPlaneCount planes worth of data is allocated in the data
20// array ( allocatedPlaneCount * 128 bytes )
21// 4. If a plane index for a layer is < 128, then the data for that layer is at
22// data[ index * 128 ]
23// 5. If a plane index for a layer is 128, then all values for that plane are 0
24
25// This class needs to be thread safe as there are times where chunk data is
26// shared between server & main threads. Data values are queried very regularly
27// so this needs to be as light-weight as possible.
28
29// To meet these requirements, this class is now implemented using a lock-free
30// system, implemented using a read-copy-update (RCU) type algorithm. Some
31// details...
32
33// (1) The storage details for the class are now packed into a single int64_t,
34// which contains both a pointer to the data that is required and a count of how
35// many planes worth
36// of storage are allocated. This allows the full storage to be updated
37// atomically using compare and exchange operations (implemented with
38// InterlockedCompareExchangeRelease64).
39// (2) The data pointer referenced in this int64_t points to an area of memory
40// which is 128 + 128 * plane_count bytes long, where the first 128 bytes stoere
41// the plane indices, and
42// the rest of the data is variable in size to accomodate however many
43// planes are required to be stored
44// (3) The RCU bit of the algorithm means that any read operations don't need to
45// do any checks or locks at all. When the data needs to be updated, a copy of
46// it is made and updated,
47// then an attempt is made to swap the new data in - if this succeeds then
48// the old data pointer is deleted later at some point where we know nothing
49// will be reading from it anymore. This is achieved by putting the delete
50// request in a queue which means it won't actually get deleted until 2 game
51// ticks after the last time its reference existed, which should give us a
52// large margin of safety. If the attempt to swap the new data in fails,
53// then the whole write operation has to be attempted again - this is the
54// only time there is really a high cost for this algorithm and such write
55// collisions should be rare.
