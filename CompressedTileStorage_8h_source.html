13// This class is used for the compressed storage of tile data. Unlike the
14// SparseLightingStorage class, data is split into 512 blocks of 4x4x4 tiles.
15// Then within each block, the data is compressed as described below, with a
16// selection of bits per tile available, in a method similar to a palettised
17// image.
18
19// There are two elements to the storage... an index array (fixed size), and the
20// actual storage required...
21
22// The index:
23// (1) Each index takes up 2 bytes, one for each 4x4x4 block ie 512 X 2 = 1024
24// bytes in total (2) The type of index is determined by the least significant 2
25// bits, the other 14 bits represent an offset for the data, stored divided by 2
26// 0 - the data for this block is represented at 1 bit per
27// tile. Data pointed to is 2 bytes describing the 2 possible tiles stored in
28// this block, followed by 32 bytes of data (total 34 bytes)
29// 1 - the data for this block is represented at 2 bit per tile. Data pointed to
30// is 4 bytes describing the 4 possible tiles stored in this block, followed by
31// 64 bytes of data (total 68 bytes) 2 - the data for this
32// block is represented at 4 bit per tile. Data pointed to is 16 bytes
33// describing the 16 possible tiles stored in this block, followed by 128 bytes
34// of data (total 144 bytes) 3 - if bit 2 is 0, then this block is
35// represented at 8 bits per tile. Data pointed to is 64 bytes, offset must be a
36// multiple of 4 (since bit 2 can also be thought of as being
37// the low bit of the offset (divided by 2 as in the other cases), and is zero)
38// - if bit 2 is 1, then this block is represented at 0
39// bits per tile. The upper 8 bits of the index store the tile value that is
40// used by the entire block.
41// So:
42// oooooooooooooo00 - 1 bit per tile, offset oooooooooooooo0
43// oooooooooooooo01 - 2 bits per tile, offset oooooooooooooo0
44// oooooooooooooo10 - 4 bits per tile, offset oooooooooooooo0
45// ooooooooooooo011 - 8 bits per tile, offset ooooooooooooo00
46// tttttttt-----111 - 0 bits per tile - tile is tttttttt
47
48// Some notes on the logic of all of this...
49// (1) Large numbers of blocks in the world really don't need to be stored at a
50// full 8 bits per tile. In a worst-case scenario, all planes would be 256 bytes
51// and we'd have to store offsets of up to 32704 ( 64 x 511). This would require
52// 15 bits per offset to store, but since in all cases the data can be stored
53// with a 2 byte alignment, we can store offsets divided by 2, freeing up 2 bits
54// to store the type of index for each plane. This allows us to encode 4 types,
55// but we really have 5 types (0, 1, 2, 4 or 8 bits per tile). Since the 8-bit
56// per tile planes are likely to be very rare, we can free up an extra bit in
57// those by making their offset 4-byte aligned, and then use the extra bit to
58// determine whether its a 0 or 8-bit per tile index. In the 0 bit case, we can
59// use the bits used for the offset to store the actual tile value represented
60// throughout the plane. (2) The compression is done per 4x4x4 block rather than
61// planes like the lighting, as that gives many more regions that have a small
62// number of tile types than per plane, and can therefore be compressed using
63// less bits per tile. This is at the expense of a larger index, and more
64// overhead from storing the tile types in each block (since there are more
65// blocks than planes). However on balance this still was found to give much
66// better compression - around 12.5% vs 19% by doing things per plane. (3)
67// Another compromise is being made on how the memory is allocated. This is all
68// currently done with physical allocations to bypass the general heap manager,
69// in particular to allow the freeing of memory to actually free whole memory
70// pages cleanly rather than leaving them as managed by the heap manager. The
71// downside to this is that all storage is done in whole 4K pages. Annoyingly,
72// a lot of our compressed chunks are just on the edge of fitting in 4K, so an
73// awful lot of them end up being 8K when they are just a small amount over.
74// However, in testing absolutely no chunks were seen that got close to going
75// over 8K compressed, so doing things this way then we at least know that we
76// are reliably getting 25% compression, and freeing things up cleanly. Note:
77// see the comments on the getIndex and getBlockAndTile for an explanation of
78// how the blocks themselves are organised in terms of mapping a chunk-wide
79// x/y/z into a block and tile index.
80
